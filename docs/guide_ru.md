# Создание yak8sui: руководство для начинающих по TUI для Kubernetes на Go

> **Для кого это?** Для разработчиков, которые только начинают изучать Go.
> Этот документ объединяет всё, что лежит в основе `yak8sui` — общение с
> Kubernetes, отрисовку терминального интерфейса и архитектуру, которая всё это
> связывает, — и объясняет *почему* было принято то или иное решение, а не
> только *что* сделано.

Проект — это небольшое терминальное приложение (TUI), которое показывает списки
**подов и деплойментов** Kubernetes, позволяет **переключаться между этими
экранами** по нажатию клавиши и **переключать неймспейсы** на лету — подобно
популярному инструменту `k9s`.

## 1. Общая картина: три слоя

Мы намеренно разделили программу на три зоны ответственности. Каждая знает
только о слое, который находится непосредственно под ней:

```
cmd/yak8sui   (точка входа)   →  запустить приложение, передать конфиг
     │
pkg/ui        (отображение)   →  рисовать таблицы, обрабатывать клавиши  [знает tview]
     │
pkg/k8s       (данные)        →  получать поды/неймспейсы                [знает client-go]
     │
Kubernetes API
```

Золотое правило:

- **`pkg/k8s` никогда не импортирует `tview`** — это чистые данные, которые можно тестировать без терминала.
- **`pkg/ui` никогда не импортирует `client-go`** — он работает только с простыми структурами Go.

Именно соблюдение этих границ делает проект легко расширяемым в будущем
(например, при добавлении экрана «деплойменты»).

### Почему директория = пакет?

В Go **пакет — это директория**, а не отдельный файл. Поэтому логика подов
живёт в `pkg/k8s/`, а UI — в `pkg/ui/`. Каждая новая зона ответственности
получает свою папку. Внутри папки может быть много файлов — все они принадлежат
одному пакету.

### Почему `cmd/yak8sui/main.go`, а не `cmd/main.go`?

Имя директории становится именем бинарного файла. `cmd/yak8sui/` создаёт бинарник
с именем `yak8sui`. Папка `cmd/` — это *контейнер* для одной или нескольких
программ. Единственное жёсткое правило Go: **один пакет `main` на директорию.**
Зарезервировав `cmd/yak8sui/` сейчас, вы сможете добавить второй бинарник позже
без перестройки структуры.

### Карта файлов проекта

Вот каждый файл и за что он отвечает. Держите эту карту под рукой — остальная часть
руководства ссылается на эти файлы по именам:

```
yak8sui/
├── cmd/
│   └── yak8sui/
│       └── main.go         # точка входа: создаёт App и вызывает Run()
├── pkg/
│   ├── k8s/                # слой ДАННЫХ (пакет k8s) — знает client-go
│   │   ├── client.go       #   newClientset(): kubeconfig → clientset
│   │   ├── pods.go         #   структура PodInfo + ListPods(namespace)
│   │   ├── deployments.go  #   структура DeploymentInfo + ListDeployments(namespace)
│   │   └── namespaces.go   #   ListNamespaces()
│   └── ui/                 # слой UI (пакет ui) — знает tview
│       ├── app.go          #   структура App, New(), Run(), SetNamespace(),
│       │                   #   переключение экранов, глобальные клавиши, picker, modal()
│       ├── pods.go         #   newPodsTable(): таблица подов + refreshData()
│       ├── deployments.go  #   newDeploymentsTable(): таблица деплойментов
│       └── colors.go       #   statusColor(): сопоставляет статус пода с цветом
├── go.mod                  # имя модуля + версии зависимостей
└── go.sum                  # криптографические контрольные суммы зависимостей
```

Как это читать: **одна папка на пакет**, **один файл на зону ответственности**.
Именно так и был добавлен экран деплойментов — файл `pkg/k8s/deployments.go`
(данные) и `pkg/ui/deployments.go` (экран), **без изменений** в существующих
файлах подов.

## 2. Слой данных: общение с Kubernetes (`client-go`)

Всё взаимодействие с Kubernetes изолировано в `pkg/k8s`. UI вызывает простые
функции вроде `ListPods(namespace)` и получает обратно обычные структуры Go.

### 2.1 Две библиотеки, две задачи

Чтобы общаться с Kubernetes, нужны два разных внешних пакета с чётко
разделёнными обязанностями:

| Пакет | Роль | Аналогия |
|---|---|---|
| `k8s.io/apimachinery` | Фундамент и правила. Описывает, как выглядят объекты (например, `metav1.ListOptions`). Ничего не знает о сети. | Правила заполнения бланка заказа |
| `k8s.io/client-go` | Транспорт и сеть. Читает `~/.kube/config`, авторизуется, отправляет HTTP-запросы. | Курьер, который везёт бланк |

Друг без друга они не работают.

### 2.2 Цепочка вызовов `client-go`

Основа любого запроса — одна длинная цепочка методов:

```go
clientset, err := kubernetes.NewForConfig(config)
pods, err := clientset.CoreV1().Pods(namespace).List(context.Background(), metav1.ListOptions{})
```

Читаем слева направо:

- **`clientset`** — главный «пульт управления», содержащий мини-клиентов для всех типов ресурсов.
- **`.CoreV1()`** — выбирает группу API `Core` версии `v1` (где живут поды, сервисы, ноды).
- **`.Pods(namespace)`** — сужает область до конкретного неймспейса.
- **`.List(...)`** — финальное действие, которое реально отправляет HTTP-запрос к API-серверу.

### 2.3 `context.Background()`

Пакет `context` — стандартный для Go механизм управления жизненным циклом и
отменой сетевых операций. Если API-сервер зависнет, контекст позволит прервать
операцию по тайм-ауту, а не зависнуть навсегда.

`context.Background()` — это абсолютно пустой контекст по умолчанию: без
тайм-аута, его нельзя отменить. Мы передаём его как «заглушку», когда нам не
нужна продвинутая отмена. (Естественный следующий шаг в обучении — заменить его
на контекст с тайм-аутом.)

### 2.4 Возвращаем структурированные данные, а не просто строки

Функция не должна возвращать голый `[]string` из имён, если вызывающему нужно
больше. Вместо этого мы определяем небольшую структуру на уровне пакета:

```go
type PodInfo struct {
    Name   string
    Status string
}

func ListPods(namespace string) ([]PodInfo, error) { ... }
```

**Структура (struct)** — это пользовательский составной тип в Go: он объединяет
логически связанные переменные (здесь — имя пода и его статус) в одну «коробку».
В отличие от классов в ООП, здесь нет скрытой магии и классического наследования
— методы привязываются к структурам извне.

### 2.5 Бонус: встраивание структур (почему работает `pod.Name`)

Когда вы отлаживаете `v1.Pod`, вы видите только `TypeMeta`, `ObjectMeta`, `Spec`
и `Status` — но в коде пишете `pod.Name`. Это **встраивание структур (struct
embedding)**:

```go
type Pod struct {
    metav1.TypeMeta   `json:",inline"`
    metav1.ObjectMeta `json:"metadata,omitempty"`  // Name находится здесь
    Spec              PodSpec
    Status            PodStatus
}
```

У первых двух полей нет имени — это **анонимные (встроенные) поля**. Go
автоматически **продвигает (promotes)** их внутренние поля на верхний уровень,
поэтому оба варианта легальны:

1. Полный путь: `pod.ObjectMeta.Name`
2. Сокращённый: `pod.Name`

> Отладчик показывает *физическое* расположение в памяти, поэтому чтобы найти
> `Name`, нужно раскрыть `ObjectMeta`. Если бы две встроенные структуры имели
> поле с одинаковым именем, Go выдал бы ошибку `ambiguous selector` на `pod.Name`,
> и пришлось бы писать полный путь.

### 2.6 Общий конструктор клиента — и паттерн обработки ошибок в Go

И `ListPods`, и `ListNamespaces` нуждаются в `clientset`, поэтому настройка
вынесена в один приватный помощник `newClientset()` в файле `client.go`:

```go
func newClientset() (*kubernetes.Clientset, error) {
    home, err := os.UserHomeDir()
    if err != nil {
        return nil, fmt.Errorf("failed to get home dir: %w", err)
    }

    kubeconfig := filepath.Join(home, ".kube", "config")

    config, err := clientcmd.BuildConfigFromFlags("", kubeconfig)
    if err != nil {
        return nil, fmt.Errorf("failed to build kubeconfig: %w", err)
    }

    clientset, err := kubernetes.NewForConfig(config)
    if err != nil {
        return nil, fmt.Errorf("failed to create kubernetes client: %w", err)
    }

    return clientset, nil
}
```

Вынос этого кода сюда избавляет `ListPods` и `ListNamespaces` от повторения
настройки. А ещё он показывает **идиому обработки ошибок в Go**, которую вы будете
видеть повсюду:

- В Go нет исключений. Функция, которая может завершиться неудачей, возвращает
  `error` **последним возвращаемым значением**; `nil` означает «успех, ошибки нет».
- После каждого «падучего» вызова вы проверяете `if err != nil` и **выходите
  досрочно**. Поэтому Go-код читается как плоская последовательность «сделай шаг —
  проверь ошибку», а не как вложенные try/catch.
- `fmt.Errorf("...: %w", err)` **оборачивает** исходную ошибку человекочитаемым
  сообщением. Глагол `%w` сохраняет исходную ошибку внутри, так что вызывающий код
  может и прочитать ваше сообщение, и докопаться до первопричины.
- При успехе последняя строка возвращает реальное значение и `nil`.

## 3. Слой UI: отрисовка с помощью `tview`

### 3.1 Почему `tview` (а не Bubble Tea)?

Для новичка в Go решающее значение имеет порог входа:

- **Bubble Tea** использует реактивную архитектуру Elm (Model-View-Update). Она
  концептуально прекрасна, но заставляет управлять потоками сообщений (`Msg`) и
  команд (`Cmd`) с самого начала, что перегружает новичков.
- **`tview`** предлагает классический императивный подход в стиле LEGO: вы
  создаёте готовые виджеты (таблицы, списки, сетки), настраиваете их свойства
  «сверху вниз» и монтируете. Код остаётся линейным и простым для отладки.

### 3.2 Строительные блоки

| Виджет | Для чего |
|---|---|
| `tview.NewTable()` | Интерактивный список подов с выбором строк |
| `tview.NewTextView()` | ASCII-арт в шапке, панель неймспейса/статуса и подсказки в подвале |
| `tview.NewGrid()` | Размещение шапки / таблицы / подвала по строкам и колонкам |
| `tview.NewList()` | Всплывающий список выбора неймспейса |
| `tview.NewPages()` | Наложение попапа *поверх* основного экрана |

### 3.3 Обновление данных через замыкание

Вместо однократной отрисовки таблицы логика заполнения строк обёрнута в
замыкание `refreshData()`. Каждый вызов:

1. Обновляет заголовок таблицы на текущий неймспейс.
2. Очищает старые строки (оставляя шапку) через `table.RemoveRow()`.
3. Запрашивает свежий срез подов из `pkg/k8s`.
4. Заново создаёт ячейки, применяя **цветовую индикацию по статусу** через
   вспомогательную функцию `statusColor` из `colors.go` (Running/Succeeded →
   зелёный, Pending → жёлтый, иначе красный).

### 3.4 Обработка ввода с клавиатуры

`SetInputCapture` задаёт реакции на события клавиш. На таблице подов:

- `Esc` / `Ctrl+C` → `app.Stop()` корректно завершает работу.
- `r` / `R` → повторно запускает `refreshData()` для ручного обновления.

## 4. Рефакторинг: `App` как единый источник истины

Это самая важная архитектурная идея проекта, поэтому разберём её медленно.

### 4.1 Проблема

До этого рефакторинга каждый экран получал неймспейс **как строку, скопированную
в его замыкание** (это *старая* сигнатура — сейчас её в коде нет, мы объясняем,
от чего мы ушли):

```go
func newPodsTable(app *tview.Application, namespace string) *tview.Table {
    // `namespace` — приватная копия, замороженная навсегда
}
```

Это нормально для *одного* статичного экрана. Но как только вы захотите
**переключать неймспейсы так, чтобы обновлялись все панели**, копии становятся
ловушкой:

- У таблицы подов своя копия.
- У будущей таблицы деплойментов будет *ещё одна* копия.
- Никто не разделяет состояние, поэтому изменение одного не обновит остальных.

### 4.2 Решение: централизованное состояние + наблюдатели

Мы вводим структуру `App`, которая **владеет** общим состоянием (текущим
неймспейсом *и* тем, какой экран активен) и списком коллбэков «обнови меня». Это
классический **паттерн «наблюдатель» (observer)** — та же идея, что лежит в основе
Redux/MVU во фронтенде.

Сначала два небольших вспомогательных типа. Так как теперь экранов больше одного,
мы помечаем каждый экран типом `ScreenType` и снабжаем каждый refresher меткой
экрана, к которому он относится:

```go
type ScreenType int

const (
    ScreenPods ScreenType = iota
    ScreenDeployments
)

// Каждый зарегистрированный refresher помнит, к какому экрану он относится.
type RefreshRequest struct {
    Screen    ScreenType
    Refresher func()
}
```

Теперь сама структура `App`:

```go
// App — единый источник истины для общего состояния UI (текущий неймспейс и то,
// какой экран активен). Экраны читают это состояние через него и подписываются на
// изменения, чтобы видимая панель оставалась синхронной.
type App struct {
    app         *tview.Application
    pages       *tview.Pages
    mainGrid    *tview.Grid
    headerLeft  *tview.TextView
    headerRight *tview.TextView

    podsTable        *tview.Table
    deploymentsTable *tview.Table

    namespace    string           // ← единственная копия
    activeScreen ScreenType       // ← Pods или Deployments
    refreshers   []RefreshRequest // ← функция перерисовки каждого экрана + метка
}
```

Методы, заставляющие паттерн работать:

```go
// Экраны ЧИТАЮТ неймспейс через это — никогда не через свою копию.
func (a *App) Namespace() string { return a.namespace }

// Экран ПОДПИСЫВАЕТСЯ, передавая свою функцию перерисовки с меткой экрана.
func (a *App) register(screen ScreenType, refresh func()) {
    a.refreshers = append(a.refreshers, RefreshRequest{Screen: screen, Refresher: refresh})
}

// Смена неймспейса перерисовывает только тот экран, что сейчас виден.
func (a *App) SetNamespace(ns string) {
    a.namespace = ns
    a.updateHeader()
    a.refreshActiveOnly()
}

// refreshActiveOnly запускает функции перерисовки только активного экрана.
func (a *App) refreshActiveOnly() {
    for _, req := range a.refreshers {
        if req.Screen == a.activeScreen {
            req.Refresher()
        }
    }
}
```

> **Почему обновляем только активный экран?** В каждый момент видна лишь одна
> таблица (поды *или* деплойменты). Перерисовка скрытой была бы напрасной работой
> *и* лишним сетевым запросом к Kubernetes за данными, на которые никто не смотрит.
> Поэтому каждый refresher помечен своим экраном, а `refreshActiveOnly` пропускает
> остальные. При переключении экранов оно тоже вызывает `refreshActiveOnly`, так
> что только что показанная таблица загружает свежие данные по требованию.

Представьте это как «звезду»: при старте каждый экран **подписывается** (с меткой
экрана), а позже один вызов `SetNamespace` **рассылает обновление — но только
refresher'ам активного экрана**:

```
   ПОДПИСКА  (каждый экран при старте вызывает a.register(ЭКРАН, refreshData))

   ┌────────────────┐        ┌──────────────────────┐
   │ таблица подов  │        │ таблица деплойментов │
   │ {Pods, refresh}│        │ {Deployments,refresh}│
   └───────┬────────┘        └──────────┬───────────┘
           │                            │
           └─────────────┬──────────────┘
                         ▼
             ┌────────────────────────┐
             │          App           │
             │  namespace             │
             │  activeScreen = Pods   │
             │  refreshers            │
             │    []RefreshRequest    │
             └───────────┬────────────┘

   РАССЫЛКА  (SetNamespace запускает лишь refresher'ы, где Screen == activeScreen)

                         │  SetNamespace("kube-system")
                         │  → refreshActiveOnly()
                         ▼
          ╔═════════════════════════════════════════╗
          ║ запускается refresh() только АКТИВНОГО   ║
          ║ экрана (скрытая таблица пропускается)    ║
          ╚═════════════════════════════════════════╝
```

### 4.3 Как экран подключается

Теперь экран принимает `*App` вместо строки `namespace`. Он **читает** неймспейс
в реальном времени и **регистрирует** себя, чтобы перерисовываться при каждом
изменении:

```go
func newPodsTable(a *App) *tview.Table {
    refreshData := func() {
        // всегда запрашивает ТЕКУЩИЙ неймспейс, никогда не устаревшую копию
        pods, err := k8s.ListPods(a.Namespace())
        // ...заполнить строки...
    }

    a.register(ScreenPods, refreshData) // подписка (с меткой экрана Pods)
    refreshData()                       // отрисовать сразу один раз
    // ...
}
```

### 4.4 Глобальные привязки клавиш

Клавиши должны работать независимо от того, какая панель в фокусе, поэтому они
перехватываются один раз — на контейнере `Pages`, который держит всё, а не на
отдельном виджете:

```go
a.pages.SetInputCapture(func(ev *tcell.EventKey) *tcell.EventKey {
    switch ev.Rune() {
    case 'p', 'P':
        a.switchToPods()
        return nil
    case 'd', 'D':
        a.switchToDeployments()
        return nil
    case 'n', 'N':
        if name, _ := a.pages.GetFrontPage(); name == "main" {
            a.showNamespacePicker()
        }
        return nil
    case 'r', 'R':
        a.refreshActiveOnly()
        return nil
    }
    return ev
})
```

- `p` / `d` — переключение между экранами подов и деплойментов.
- `n` — открыть выбор неймспейса (только когда впереди страница `"main"`, чтобы
  попап не открывался поверх самого себя).
- `r` — вручную обновить активный экран.

### 4.5 Переключение экранов

Поскольку обе таблицы создаются один раз и хранятся в `App` (`podsTable`,
`deploymentsTable`), переключение экрана — это просто подмена таблицы в средней
строке сетки:

```go
func (a *App) switchToDeployments() {
    if a.activeScreen == ScreenDeployments {
        return // уже здесь, ничего делать не нужно
    }
    a.activeScreen = ScreenDeployments
    a.updateHeader()

    a.mainGrid.RemoveItem(a.podsTable)
    a.mainGrid.AddItem(a.deploymentsTable, 1, 0, 1, 1, 0, 0, true)

    a.app.SetFocus(a.deploymentsTable)
    a.refreshActiveOnly() // загрузить свежие данные для показанного экрана
}
```

`switchToPods` — зеркальная копия. Каждый начинается с проверки «я уже здесь?»,
чтобы повторное нажатие той же клавиши ничего не делало.

### 4.6 Попап через `Pages`

`tview.Pages` позволяет накладывать экраны друг на друга. Список выбора
неймспейса — это `tview.List`, полученный из кластера и отцентрированный с
помощью небольшого помощника `modal()`, который использует пустые flex-элементы
как распорки. Выбор пункта вызывает `a.SetNamespace(...)` и удаляет страницу.

### 4.7 Почему это масштабируется

| Задача | Как решена |
|---|---|
| Все панели используют один неймспейс | Они вызывают `a.Namespace()`, а не копию |
| Смена неймспейса перерисовывает экран | `SetNamespace` → `refreshActiveOnly()` |
| Работает только видимый экран | Refresher'ы помечены типом `ScreenType` |
| Переключение между экранами | `p` / `d` подменяют таблицу в сетке |
| Работает из любой панели в фокусе | Клавиши перехвачены на `pages`, а не на виджете |
| Добавление ещё одного экрана | Создать таблицу + `a.register(screen, refresh)` |

Экран деплойментов *действительно* был добавлен по этому рецепту — это больше не
гипотетический пример:

```go
func newDeploymentsTable(a *App) *tview.Table {
    refreshData := func() { /* k8s.ListDeployments(a.Namespace()) ... */ }
    a.register(ScreenDeployments, refreshData) // метка экрана Deployments
    refreshData()
    // ...
}
```

## 5. Механика Go вблизи

Этот раздел подробно разбирает возможности языка Go, благодаря которым работает
описанная выше архитектура. Если вы новичок в Go — это те самые концепции, после
которых наступает «ага, понятно!».

> **Не знакомы с указателями?** Несколько символов встречаются по всему коду:
> - `*App` означает «указатель на `App`» — стрелку, направленную на реальную
>   структуру в памяти, а не её копию.
> - `&App{...}` означает «создай `App` и дай мне его адрес» (указатель на него).
> - `a.namespace` читает поле *через* указатель — Go сам идёт по стрелке, никакого
>   особого синтаксиса не нужно.
>
> Зачем это, а не просто копировать структуру? Две причины: (1) копировать большую
> структуру при каждом вызове расточительно, и (2) что важнее, если бы у каждого
> экрана была своя *копия* `App`, изменение `namespace` в одной не затронуло бы
> остальные. Вся архитектура держится на **одном общем `App`** — указатели
> гарантируют, что все смотрят на одну и ту же коробку в памяти.

### 5.1 Методы и получатели (receivers): как поведение привязывается к данным

В ООП-языках (Java, C#, Python) методы пишут *внутри* тела класса. Go физически
разделяет данные и поведение, а затем связывает их через **получатель (receiver)**.

Структура хранит только данные:

```go
type App struct {
    app   *tview.Application
    pages *tview.Pages
    // ...
}
```

Метод — это обычная функция с дополнительным `(a *App)` перед именем:

```go
func (a *App) Run() error {
    // ...
}
```

Это `(a *App)` и есть «клей». Это **получатель-указатель (pointer receiver)**, и он
говорит Go: «функция `Run` принадлежит структуре `App`, а внутри неё `a` — это тот
конкретный экземпляр, на котором её вызвали». Поэтому когда в `main.go` написано:

```go
app := ui.New(ui.Config{Namespace: "kube-system"})
err := app.Run()
```

Go подставляет вашу переменную `app` внутрь `Run` под именем `a`. Внутри `Run` вы
можете писать `a.app.SetRoot(...)` или `a.showNamespacePicker()`.

> **Почему указатель (`*App`), а не значение (`App`)?** Получатель-указатель
> позволяет методу *изменять* структуру — `SetNamespace` должен перезаписать
> `a.namespace`, а это работает только через указатель. Кроме того, так структура
> не копируется целиком при каждом вызове. Правило: если в структуре больше
> нескольких полей или хотя бы один метод её меняет — используйте
> получатель-указатель.
>
> **Почему `App` с заглавной, а `namespace` со строчной:** В Go идентификатор,
> начинающийся с **заглавной буквы, экспортируется** (он публичный — доступен из
> других пакетов); **строчная буква — не экспортируется** (приватный, виден только
> внутри своего пакета). Поэтому `App`, `Run` и `Namespace` публичные, а
> `namespace` и `register` приватны для `pkg/ui`. Именно поэтому экраны должны
> читать неймспейс через метод `Namespace()`, а не лезть в `a.namespace` напрямую.

Практическая польза: **методы одной структуры могут лежать в разных файлах** одного
пакета. Любая функция, которая начинается с `(a *App)`, приклеивается к той же
структуре `App`, поэтому их можно разнести по файлам, а не раздувать один. Именно
поэтому `Run`, `SetNamespace` и `showNamespacePicker` — все методы `App`.

### 5.2 Паттерн «Конструктор»: `New` + `Config`

В Go нет встроенных конструкторов, поэтому по соглашению пишут функцию `New`.
Проследим, как строка `"kube-system"` путешествует из `main.go` в работающее
приложение:

```
1. В main.go:                "kube-system"
      ▼
2. Упаковали в структуру:    ui.Config{Namespace: "kube-system"}
      ▼
3. Передали как аргумент:     ui.New( ui.Config{...} )
      ▼
4. Прилетело в New() в параметр: cfg
      ▼
5. Переложили в приватное поле структуры: a.namespace
      ▼
6. Теперь Run() всегда знает текущий неймспейс
```

Принимающая сторона:

```go
func New(cfg Config) *App {
    return &App{
        app:       tview.NewApplication(),
        namespace: cfg.Namespace, // ← вот она, связь
    }
}
```

Зачем оборачивать аргументы в структуру `Config`, а не передавать их напрямую? Это
позволяет безболезненно добавлять будущие опции (имя контекста, интервал
обновления…), не меняя сигнатуру `New`. Связь при этом **строгая**: опечатка вроде
`ui.Config{NameSpace: ...}` (заглавная S) или несуществующее поле не скомпилируются
— Go проверяет имена полей до буквы.

> **`:=` против `=`:** Оператор `:=` (в `main.go` это `app := ui.New(...)`)
> объявляет новую переменную *и* выводит её тип из значения справа. Обычный `=`
> используется только для переприсваивания *уже объявленной* переменной. Если бы
> вы написали `var app *ui.App` отдельной строкой, следующей было бы
> `app = ui.New(...)`. Также отметьте `&App{...}` — `&` берёт адрес литерала
> структуры, поэтому `New` возвращает указатель (`*App`), а не копию.

### 5.3 Что лежит в памяти и бесконечный цикл `Run`

После старта в памяти «висит» ровно одна большая коробка — переменная `app` типа
`*ui.App`:

```
[ app (*ui.App) в оперативной памяти ]
├── namespace        = "kube-system"
├── activeScreen     = ScreenPods
├── app              = [запущенный tview.Application]
├── pages            = [менеджер экранов, сейчас держит главный Grid]
├── mainGrid         = [шапка / активная таблица / подвал]
├── headerLeft       = [ASCII-арт заголовок]
├── headerRight      = [статус: активный экран + неймспейс]
├── podsTable        = [виджет таблицы подов]
├── deploymentsTable = [виджет таблицы деплойментов]
└── refreshers       = [{Pods, refreshData}, {Deployments, refreshData}]
```

Когда в `main.go` выполняется `app.Run()`, управление уходит *внутрь* `tview` и
**замирает там**. Программа входит в цикл событий (Event Loop) — представьте
охранника в бесконечном `for {}`, который ждёт сигналов:

1. **Спит и ждёт** — почти не нагружает процессор, просто ждёт событий терминала.
2. **Ловит нажатие** — вы нажимаете `n`.
3. **Будит перехватчик ввода** — `tview` дёргает функцию, которую мы привязали в `app.go`.
4. **Меняет картинку** — обработчик говорит `a.pages` показать список неймспейсов поверх.
5. **Снова засыпает** — до вашего следующего нажатия или клика.

Это классическое **событийно-ориентированное приложение**: статичное *состояние*
(структура `App`) плюс *движок* (`Run`), который оживает только на ввод, меняет
состояние и снова засыпает. `main.go` при этом просто ждёт на `app.Run()`, пока
цикл не завершится (например, через `a.app.Stop()`).

### 5.4 Функции как значения (и замыкания)

В Go функция — это значение, как строка или число: её можно сохранить в
переменную, положить в срез или передать как аргумент. Именно это движет реестром
наблюдателей.

Представьте `App` как **редакцию журнала** с пустым списком подписчиков
(`refreshers []RefreshRequest`). Таблица подов — это **читатель**. Когда она
вызывает `a.register(ScreenPods, refreshData)`, она не просит данные прямо сейчас
— она отдаёт свою «визитку» (функцию плюс метку, для какого она экрана), чтобы её
вызвали позже:

```go
type RefreshRequest struct {
    Screen    ScreenType
    Refresher func() // ← ФУНКЦИЯ, хранимая как значение
}

func (a *App) register(screen ScreenType, refresh func()) {
    a.refreshers = append(a.refreshers, RefreshRequest{Screen: screen, Refresher: refresh})
}
```

Тип `func()` означает «функция без аргументов и без возвращаемого значения».
Важно: мы передаём `refreshData` **без** круглых скобок — `register(..., refreshData)`
отдаёт саму функцию; а `refreshData()` *вызвал* бы её и передал результат. Позже
`refreshActiveOnly` запускает подходящие **со** скобками:

```go
for _, req := range a.refreshers {
    if req.Screen == a.activeScreen {
        req.Refresher() // скобки () означают «выполни сейчас»
    }
}
```

Вся магия в том, что `refreshData` — это **замыкание**: оно «захватило» переменную
`table` из `newPodsTable`. Даже после того, как мы отдали его в `App`, оно помнит,
какую именно таблицу очищать и заполнять. При этом `App` вообще не знает, что
существует файл `pods.go` — он просто вызывает абстрактные функции. Зарегистрируйте
ещё таблицы (деплойменты, сервисы…) — и тот же цикл обновит тот экран, что активен.

```
   newPodsTable(a *App)                          App
   ┌──────────────────────────────┐             ┌──────────────────┐
   │ table := tview.NewTable() ◀┐ │             │                  │
   │                            │ │             │ refreshers       │
   │ refreshData := func() {    │ │             │   []RefreshRequest│
   │     table.SetTitle(...) ───┘ │  захват     │                  │
   │     pods := k8s.ListPods(...)│             │                  │
   │     ...заполняем строки...   │             │                  │
   │ }                            │             │                  │
   │                              │             │                  │
   └──────────┬───────────────────┘             │                  │
              │  a.register(ScreenPods, refreshData)              │
              └─────────────────────────────────▶  лежит здесь    │
                                               └──────────────────┘

   Позже App вызывает refreshData() из refreshActiveOnly. Замыкание ВСЁ ЕЩЁ
   помнит `table`, поэтому знает, какую таблицу очистить и заполнить, —
   хотя сам App никогда не видел файл `pods.go`.
```

### 5.5 Разбор: окно выбора неймспейса и трюк с центрированием

`showNamespacePicker` (в `app.go`) строит попап:

```go
list := tview.NewList().ShowSecondaryText(false)         // компактный список
list.SetBorder(true).SetTitle(" Select namespace (Esc to cancel) ")

namespaces, err := k8s.ListNamespaces()                  // спрашиваем кластер
```

При ошибке он добавляет один пункт с текстом ошибки; иначе — по одному пункту на
неймспейс, каждый со своим коллбэком, который срабатывает по Enter:

```go
for _, ns := range namespaces {
    ns := ns // захватываем значение этой итерации (см. «Подводные камни»)
    list.AddItem(ns, "", 0, func() {
        a.SetNamespace(ns)                // запускает всю цепочку обновлений
        a.pages.RemovePage("namespace")   // закрываем попап
    })
}
```

`Esc` удаляет страницу, ничего не меняя. Наконец, список монтируется, и фокус
клавиатуры переводится на него, чтобы стрелки управляли списком, а не таблицей под
ним:

```go
a.pages.AddPage("namespace", modal(list, 40, 20), true, true)
a.app.SetFocus(list)
```

**Трюк с центрированием (`modal`):** в терминале нет готовой функции «разместить
окно по центру». `modal` вкладывает два flex-контейнера друг в друга и использует
`nil`-элементы как невидимые распорки — по одной слева и справа по горизонтали и по
одной сверху и снизу по вертикали, — удерживая блок фиксированного размера (40×20)
ровно по центру при любом изменении размера терминала.

## 6. Всё вместе: как происходит переключение неймспейса

Вот полная цепочка событий, когда пользователь нажимает `n` и выбирает неймспейс:

```
Пользователь нажимает 'n'
   │
   ▼
Срабатывает pages.SetInputCapture  →  a.showNamespacePicker()
   │
   ▼
k8s.ListNamespaces()  →  попап tview.List с неймспейсами
   │
   ▼
Пользователь выбирает "kube-system"  →  a.SetNamespace("kube-system")
   │
   ├─ a.namespace = "kube-system"      (обновить единый источник истины)
   ├─ a.updateHeader()                 (в шапке: активный экран + неймспейс)
   └─ a.refreshActiveOnly()            (перерисовывается только ВИДИМЫЙ экран)
          │
          ▼
      активная таблица вызывает k8s.List…(a.Namespace())  →  появляются новые строки
```

Одно изменение состояния — и видимый экран остаётся синхронным. Переключение
экранов через `p` / `d` работает так же: меняет `activeScreen`, подменяет таблицу
в сетке и вызывает `refreshActiveOnly()`, чтобы показанная таблица загрузилась по
требованию.

## 7. Подводные камни

### Ошибка `terminal entry not found: term not set` в IDE
Запуск TUI из консоли отладки GoLand/VSCode может приводить к падению, потому что
эта консоль не сообщает тип терминала.
**Решение:** задайте переменную окружения `TERM=xterm-256color` в конфигурации
запуска и включите опцию «Emulate terminal in output console».

### Исчезающий текст `[r]` в подвале
При `SetDynamicColors(true)` текст вроде `[r]` воспринимается как *цветовой тег* (и
исчезает). Решение — экранировать литеральную скобку, удвоив её: `[[` отрисуется
как одна `[`. Именно поэтому подвал в `app.go` пишет строки вроде
`[[[yellow]n[white]]]amespace`: ведущие `[[` печатают настоящую `[`, а затем
`[yellow]`/`[white]` переключают цвет, подсвечивая клавишу-сокращение.

### Захват переменной цикла в замыкании
Когда вы создаёте замыкание внутри цикла `for` и оно использует переменную цикла,
нужно убедиться, что каждое замыкание захватывает значение *именно этой итерации*,
а не одну общую переменную:

```go
for _, ns := range namespaces {
    ns := ns // копия для этой итерации
    list.AddItem(ns, "", 0, func() { a.SetNamespace(ns) })
}
```

**История:** В версиях Go до 1.22 `ns` была *одной* переменной, переиспользуемой
каждую итерацию. Все замыкания ссылались на неё, и к моменту их запуска (по Enter)
цикл уже закончился, а `ns` держала *последнее* значение — все пункты переключали
бы на последний неймспейс. Строка `ns := ns` затеняла её свежей копией для каждой
итерации и чинила это.

**Начиная с Go 1.22** (в этом проекте 1.26) переменные цикла по умолчанию
создаются заново каждую итерацию, так что `ns := ns` здесь технически уже не нужна.
Мы оставляем её, потому что она безвредна, делает намерение очевидным для тех, кто
учился на старых версиях Go, и это распространённый приём, который вы встретите в
реальных кодовых базах.
